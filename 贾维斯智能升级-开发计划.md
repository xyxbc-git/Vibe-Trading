### 核心缺陷（为什么「又慢又蠢」）

| # | 缺陷 | 现状 | 影响 |
| --- | --- | --- | --- |
| 1 | **无行情分类器** | 一套策略打天下 | 震荡吃亏、趋势也吃亏，哪种行情都不稳 |
| 2 | **单层信号结构** | 入场条件 AND/OR + 固定止盈止损 | 无法实现马丁均价、利润对冲、趋势过滤 |
| 3 | **止损止盈写死** | `sl_pct=0.015, tp_pct=0.025` 硬编码 | 不随波动率变化，小波动止损太松、大波动太紧 |
| 4 | **无实盘反馈闭环** | 进化引擎只在回测层面迭代 | 实盘表现不好也不会自动触发重新进化 |
| 5 | **无可视化反馈** | 桌面终端页面写了但未验证可跑 | 用户看不到系统在做什么，信任感为零 |
| 6 | **无多时间框架** | 只看 15m 一个级别 | 缺乏大趋势判断（1h/4h 方向）|
| 7 | **无自主学习** | LLM 文本层面的「教训」 | 不会从数据中发现规律，只是换因子组合 |

> **目标**：从「一套参数打天下」升级为「识别行情 + 匹配策略 + 多层防御」

### P2-01 · 行情分类器（Market Regime Classifier）

**优先级**：P0
**工期**：2-3 天
**新增文件**：`jarvis_regime_classifier.py`

**三种行情类型**：

| 类型 | 特征 | 判定规则 |
| --- | --- | --- |
| **震荡（Range）** | 价格在通道内反复 | ADX < 25 且布林带宽收窄 |
| **单边趋势（Trend）** | 价格持续沿一个方向 | ADX > 30 且 EMA20/50/200 三线共振 |
| **突破（Breakout）** | 价格从震荡突然拉出 | 布林带突破 + 成交量放大 > 2倍 |

**分类器设计**：

```python
class RegimeClassifier:
    def classify(self, klines_15m, klines_1h, klines_4h) -> dict:
        """多时间框架行情分类。
        
        Returns:
            {
                "regime": "trending" | "ranging" | "breakout",
                "confidence": 0.0~1.0,
                "direction": "bullish" | "bearish" | "neutral",
                "strength": 0.0~1.0,
                "sub_regime": "strong_trend" | "weak_trend" | ...,
                "reasoning": "三线共振看多，ADX=35..."
            }
        """
```

**关键指标**：
- ADX（平均方向指数）：趋势强度
- 布林带宽度：波动率状态
- EMA 三线排列：趋势方向
- 成交量变化率：突破确认
- ATR 变化率：波动率趋势

**验收标准**：
- [ ] 分类器能在 100 根 K 线上运行，延迟 < 100ms
- [ ] 用 2025 年 BTC 数据回测，分类准确率 > 70%
- [ ] 支持多时间框架输入（15m + 1h + 4h）

### P2-02 · 复合策略框架（Composite Strategy）

**优先级**：P0
**工期**：3-4 天
**新增文件**：`jarvis_composite_strategy.py`

**三层结构设计**：

```
┌─────────────────────────────────────────────────────┐
│  第三层：保命系统（Trend Filter）                      │
│  三均线共振确认强趋势时，抑制逆势操作                   │
│  → 震荡行情允许双向 / 趋势行情只允许顺势               │
├─────────────────────────────────────────────────────┤
│  第二层：解困系统（Hedge / Profit Offset）              │
│  遇单边行情启动利润对冲，用反向盈利单平亏损单            │
│  → 优先处理高风险头寸（亏损最大的先解决）               │
├─────────────────────────────────────────────────────┤
│  第一层：进攻系统（Core Signal + Grid/Martin）         │
│  震荡行情用均价止盈高频盈利                            │
│  趋势行情用突破追踪盈利                               │
│  → 行情分类器决定用哪套进攻逻辑                       │
└─────────────────────────────────────────────────────┘
```

**核心数据结构**：

```python
class CompositeStrategy:
    """复合策略管理器。"""
    
    def __init__(self, regime_classifier, config):
        self.classifier = regime_classifier
        self.layers = {
            "attack": AttackLayer(),      # 进攻：信号生成 + 网格/马丁
            "rescue": RescueLayer(),      # 解困：利润对冲
            "protect": ProtectLayer(),    # 保命：趋势过滤
        }
    
    def decide(self, klines, positions, balance) -> Decision:
        """综合三层给出交易决策。
        
        Returns:
            Decision(
                action="open_long" | "close_long" | "add_position" 
                       | "hedge" | "hold" | ...,
                size=..., 
                stop_loss=..., 
                take_profit=...,
                reasoning="震荡行情+RSI超卖，开多；
                           三线共振确认方向一致，保命层放行"
            )
        """
```

**进攻层细节（AttackLayer）**：

| 行情 | 策略风格 | 说明 |
| --- | --- | --- |
| 震荡 | 网格/马丁 | 区间内高频交易，加仓均价止盈 |
| 趋势 | 趋势追踪 | 突破入场，移动止盈跟踪趋势 |
| 突破 | 突破跟进 | 放量突破即入场，快速止盈 |

**马丁/网格加仓逻辑**：
```python
class MartinGrid:
    """震荡行情下的网格加仓。"""
    
    def calc_add_position(self, position, current_price, atr):
        """判断是否加仓。
        
        加仓条件：
        1. 当前持仓浮亏 > 1x ATR
        2. 加仓后总仓位不超过限额
        3. 加仓次数不超过 max_adds（默认3）
        
        加仓后更新均价，止盈按新均价 + 0.5x ATR 计算。
        """
```

**解困层细节（RescueLayer）**：
```python
class RescueLayer:
    """利润对冲解困。
    
    当持仓浮亏超过阈值，且行情分类器判定为单边趋势：
    1. 在趋势方向开反向对冲仓
    2. 对冲仓盈利后，用利润覆盖原仓亏损
    3. 先处理亏损最大的持仓
    """
```

**保命层细节（ProtectLayer）**：
```python
class ProtectLayer:
    """趋势过滤保命。
    
    规则：
    1. 三均线（EMA20/50/200）共振确认强趋势时：
       - 禁止逆势开仓
       - 已有逆势持仓发出警告
    2. ADX > 40 的强趋势中：
       - 马丁加仓被抑制（不再逆势加仓）
       - 切换为趋势追踪模式
    3. 多时间框架一致性检查：
       - 15m 信号方向 vs 1h 趋势方向
       - 不一致时降低仓位或跳过
    """
```

**验收标准**：
- [ ] 三层结构能协同决策
- [ ] 震荡行情自动切换网格/马丁模式
- [ ] 趋势行情自动抑制逆势操作
- [ ] 解困层能在模拟盘中正确执行利润对冲

### P2-03 · 动态止盈止损系统

**优先级**：P1

**修改文件**：`jarvis_scalper_trader.py`

**替换硬编码止盈止损**：

```python
class DynamicStopManager:
    """基于 ATR 的动态止盈止损。"""
    
    def calculate(self, entry_price, direction, atr, regime):
        """根据行情类型调整止损止盈倍数。
        
        | 行情 | 止损 (ATR倍数) | 止盈 (ATR倍数) | 移动止损 |
        | 震荡 | 1.0x | 1.5x | 否 |
        | 趋势 | 1.5x | 3.0x | 是（0.5x ATR 步进） |
        | 突破 | 1.2x | 2.0x | 是（0.3x ATR 步进） |
        """
    
    def trailing_stop(self, position, current_price, atr):
        """移动止损：只向有利方向移动。
        
        趋势行情下，止损随价格移动但不回退：
        - 多仓：止损 = max(当前止损, 最高价 - 1.5*ATR)
        - 空仓：止损 = min(当前止损, 最低价 + 1.5*ATR)
        """
```

**验收标准**：
- [ ] 止损止盈根据 ATR 动态计算
- [ ] 移动止损在趋势行情中正确跟踪
- [ ] 不同行情类型使用不同倍数

## 4. Phase 3 · 让他会成长（自适应闭环）

> **目标**：实盘结果自动驱动策略进化，连续亏损自动触发调整

### P3-01 · 实盘表现追踪器（Performance Tracker）

**优先级**：P0
**工期**：2 天
**新增文件**：`jarvis_performance_tracker.py`

**设计**：

```python
class PerformanceTracker:
    """实盘表现追踪与评估。
    
    持续监控：
    - 滚动 50 笔交易的胜率、盈亏比、夏普
    - 不同行情类型下的表现分类
    - 连续亏损检测
    - 策略衰退检测（性能持续下降）
    """
    
    def evaluate(self) -> PerformanceReport:
        """生成表现评估报告。"""
        
    def detect_decay(self) -> bool:
        """检测策略是否在衰退。
        
        衰退信号：
        1. 近 20 笔胜率 < 40%（对比历史胜率）
        2. 盈亏比持续下降 3 个评估周期
        3. 连续亏损 >= 5 笔
        4. 本周表现 vs 上周表现下降 > 30%
        """
    
    def trigger_re_evolve(self):
        """衰退确认后自动触发重新进化。"""
```

**自动触发机制**：
```
实盘交易 → 追踪器记录 → 评估表现
  ├─ 表现正常 → 继续交易
  ├─ 轻微衰退 → 调整参数（微调止损倍数/信号阈值）
  ├─ 严重衰退 → 触发重新进化（保留当前策略为基线）
  └─ 行情类型变化 → 自动切换策略组合
```

**验收标准**：
- [ ] 每 10 笔交易自动生成评估报告
- [ ] 连续 5 笔亏损自动触发警告
- [ ] 策略衰退检测准确率 > 80%

### P3-02 · 参数自适应引擎（Adaptive Tuner）

**优先级**：P1
**工期**：2-3 天
**新增文件**：`jarvis_adaptive_tuner.py`

**设计**：

```python
class AdaptiveTuner:
    """基于实盘数据的参数自适应。
    
    不是重新进化策略，而是微调现有策略的参数：
    - ATR 周期根据近期波动率自适应
    - 止盈止损倍数根据近期盈亏比调整
    - 信号阈值根据近期胜率微调
    - 仓位大小根据凯利公式动态计算
    """
    
    def tune(self, strategy, recent_trades, market_stats):
        """微调策略参数。
        
        约束：
        - 单次调整幅度不超过 20%
        - 调整后参数不超出合法范围
        - 每 24h 最多调整一次
        - 调整前后都要记录，支持回滚
        """
    
    def kelly_sizing(self, win_rate, avg_win, avg_loss, balance):
        """凯利公式计算最优仓位。
        
        f* = (p*b - q) / b
        p = 胜率, q = 1-p, b = 盈亏比
        实际使用半凯利（f* / 2）控制风险
        """
```

**验收标准**：
- [ ] 参数调整在合理范围内（±20%）
- [ ] 凯利公式仓位计算正确
- [ ] 调整历史可追溯，支持回滚

### P3-03 · 进化引擎 v2（智能进化）

**优先级**：P1
**工期**：2-3 天
**修改文件**：`jarvis_scalper_evolve.py`

**升级内容**：

1. **进化目标从「单策略」变为「策略组合」**：
   - LLM 生成的不再是单个策略，而是「行情A用策略X + 行情B用策略Y」的组合
   - 回测时模拟行情切换，验证组合的综合表现

2. **进化数据源增加实盘**：
   - 不只用历史回测，还参考最近 30 天实盘数据
   - 实盘中表现最差的行情类型优先进化

3. **教训系统升级**：
   - 从「文字描述教训」升级为「结构化教训」
   - 教训关联到具体行情类型和市场条件
   - LLM 能根据结构化教训精准避雷

```python
# 结构化教训格式
```

**验收标准**：
- [ ] 进化输出策略组合而非单策略

## 9. 风险与注意事项

1. **回测 ≠ 实盘**：所有策略必须先模拟盘跑 7 天再实盘
2. **过拟合风险**：进化引擎可能对历史数据过度拟合，需要 OOS 验证
3. **马丁风险**：马丁/网格策略在极端行情下可能放大亏损，必须设置最大加仓次数和总仓位上限
4. **API 限频**：Binance API 有调用频率限制，多时间框架数据要做缓存
5. **LLM 依赖**：进化引擎依赖 LLM API，需要做降级处理（API 不可用时使用最近的名人堂策略）
