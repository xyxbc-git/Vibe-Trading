## P0 · 必须修（影响认知正确性与资金安全）

### P0-1 做空持仓的浮盈亏、钱包核算、熔断判定全部方向反了

**问题**：系统多处计算盈亏时忽略持仓方向（side），做空单的数字是反的。

**证据**：
- `jarvis_paper_trader.py:922` — `stats()` 浮动盈亏 `upnl = round((price - p["entry_price"]) * p["qty"], 4)`，无方向符号。对比同文件 `_close_position` 的已实现盈亏 `jarvis_paper_trader.py:374-376` 有 `sign = -1 if open_side == "sell" else 1`——平仓算对了，持仓中是错的。
- `jarvis_paper_trader.py:391-395` — 平仓钱包回款 `proceeds = exit_price × qty` 直接入账。空单开仓冻结 `entry×qty`，价格涨到 160 平仓（亏 60%），钱包却回款 `160×qty`（比开仓还多）——**空单亏钱、钱包余额反而变多**。
- `jarvis_paper_trader.py:925,931` — 持仓市值 `holdings_value += price × qty` 计入 `equity = cash + frozen + holdings_value`，空单价格越涨「市值」越高 → `equity_usdt` 失真。
- `jarvis_circuit_breaker.py:211-217` — 单仓亏损熔断 `chg = (cur/entry - 1)×100; if chg <= -25%` 触发。**空单浮盈 25%（价格跌 25%）会误触发熔断；空单浮亏 25%（价格涨 25%）永远不触发**。
- 连锁污染：组合回撤熔断的数据源是 `jpt.stats(cfg)` 的 `equity_usdt`（`jarvis_circuit_breaker.py:196-207`），有空单在手时回撤口径整体失真。

**风险场景（大白话）**：你开了个空单，行情暴涨你在亏钱，但界面权益曲线反而往上走、熔断永远不响；直到平仓那一刻账面突然跳水。你以为自己风控在线，其实保护系统对空单是「睁眼瞎」。12 系统共识 bearish 时自动开空（`jarvis_paper_trader.py:595`），这不是边缘场景。

**改进方案**：所有浮盈亏/市值/熔断判定统一走「方向性盈亏」函数：`pnl_pct = (price/entry - 1) × 100 × (-1 if side=='sell' else 1)`；空单市值贡献改为 `2×entry×qty − price×qty`（或直接用 冻结保证金+浮动盈亏 口径）；钱包回款改为 `entry×qty + realized_pnl`。已有正确样板可抄：`jarvis_order_lifecycle.py` 的 `_direction_pnl_pct()`（任务 G 刚落地，做空镜像已测试覆盖）。

**预期效果**：空单权益/回撤/熔断全部回到真实口径，熔断不再漏报误报。 **成本：M**（改 4 处 + 补做空冒烟用例）。

### P0-2 模拟成交零手续费、零滑点、碰价即成 → 模拟胜率系统性虚高

**问题**：模拟盘的成交模型是「无摩擦真空」，用户会把虚高的胜率/盈亏比当真。

**证据**：
- 全文检索 `jarvis_paper_trader.py` 无 fee/slippage 字样（grep 零命中）——开仓、平仓、限价撮合均无手续费与滑点。
- `jarvis_paper_trader.py:806-825` — 限价买单 `price <= limit_price` 即按限价全额成交，注释自述「限价单不会比挂价更差」。真实市场 touch ≠ fill（排队深度、只打一半就弹走）。
- `jarvis_paper_trader.py:591-593,641` — 12 系统市价开仓直接按 `latest_price` 原价成交，无冲击成本。
- 滑点估计模块**已存在**：`jarvis_slippage.py:50,64,113`（`FALLBACK_BPS=10`、`estimate_slippage_pct`、`apply_fill_price`），但只在 `jarvis_executor.py:202-211` 作为**展示性预估**，不修正成交价。
- 手续费同样只有提示：`jarvis_position_calc.py:68` `TAKER_FEE_PCT = 0.05`（单边 0.05%）仅用于计算器展示「盈利额未扣手续费」。

**风险场景（大白话）**：真实合约 taker 开平双边 ≈0.1% + 滑点 ≈0.05~0.1%。一个模拟胜率 55%、平均盈利 1% 的短线策略，扣掉每笔 0.15~0.2% 摩擦后可能直接变亏损策略。你照模拟战绩上实盘，就是「按美颜照片相亲」。

**改进方案**：paper 成交统一过 `apply_fill_price()`（买加/卖减 one_way_bps），并按名义 × taker 费率记一笔 `fee` 进 wallet_ledger 与 realized_pnl；限价单成交价仍用限价但**扣费**；报表加「含费口径」标记。摩擦参数进 `jarvis_config`（默认 taker 0.05%、滑点走 jarvis_slippage）。

**预期效果**：模拟战绩≈实盘可复现战绩，胜率认知不再虚高；策略筛选门槛自动抬高。 **成本：M**。

### P0-3 止损/止盈触发邮件依赖「每单手动配置」，没配置就静默——用户以为有保护

**问题**：止损被打了，邮件不一定来。

**证据**：
- `jarvis_paper_trader.py:400-411` — 平仓仅 `reason in ("take","stop")` 时尝试邮件。
- `jarvis_order_notify.py:253-255` — `config_for_position()` 查不到该单配置 → `{"sent": False, "skipped": "该单未配置邮件提醒"}`，**没有全局收件人回退**（对比价位提醒 `jarvis_price_alert.py:733` 计划无收件人时回退全局通讯录）。
- `jarvis_paper_trader.py:401` — `time`（到期）/`signal`（信号反转）/`manual`/`limit_sell` 平仓**从不发邮件**。
- 自动跟盘动作通知走 TG/飞书：`jarvis_paper_trader.py:850-867` `_notify_actions` → `jarvis_notify`，而 `jarvis_notify.py:13,35-40` 未配 token/webhook 的渠道**静默跳过**——一个渠道都没配时等于没有任何通知。

**风险场景（大白话）**：半夜止损被打穿，你第二天中午打开电脑才知道；或者持仓被「信号反转」自动平掉了，你根本不知道仓没了。对小白来说「以为有通知」比「知道没通知」更危险。

**改进方案**：① `notify_position_closed` 在无每单配置时回退价位提醒全局通讯录（一行策略变化）；② 平仓邮件覆盖所有 reason（至少 time/signal 也发）；③ 设置页加「通知自检」卡：SMTP/TG/飞书各渠道当前是否可用，一键测试。注：任务 G 已补「入场成交邮件 + 浮盈 20%/浮亏 50% 邮件」（走全局通讯录，`jarvis_order_lifecycle.py`），入场与极端行情两头已有保底，但**止损触发本身**的盲区仍在。

**预期效果**：所有仓位状态变化都有推送兜底，消灭「静默平仓」。 **成本：S**。

### P0-4 12 系统自动跟盘绕过仓位/组合风险红线，且无冷却、无总仓上限

**问题**：手动决策路径有完整护栏，自动路径反而是「裸奔」。

**证据**：
- 护栏基准（有护栏的路径）：`open_from_decision` → `jx.evaluate_guardrails`（`jarvis_paper_trader.py:261-263`）：单笔仓位 ≤ `max_position_pct` 40%（`jarvis_executor.py:178-183`）、组合风险 ≤1.5% 自动缩仓（`jarvis_executor.py:184-191`）。
- 绕过点：`open_from_twelve` 仓位直接取信号计划 `plan.position_pct`，仅 clamp 到 0.1~100%（`jarvis_paper_trader.py:608-614`），**不经** `evaluate_guardrails`——40% 单笔上限和 1.5% 组合风险红线对它无效。
- 无总持仓数上限：`run_twelve_cycle` 对 watchlist 逐币开仓（`jarvis_paper_trader.py:656-671`），默认 7 币（`jarvis_config.py:62`）可同时 7 仓，唯一约束是钱包余额（`jarvis_paper_trader.py:631-633`）。对比 intraday 引擎有 `intraday_max_open_positions=3`（`jarvis_config.py:90`、`jarvis_intraday_trader.py:409-410`）。
- 无平仓后冷却：同币仅查「已有未平仓」（`jarvis_paper_trader.py:563-564`），止损后下一轮（15 分钟，`jarvis_dashboard.py:2791-2814`）共识若仍在会**立即重开同方向仓**——自动化的报复性交易。对比 intraday 有 `cooldown_bars`（`jarvis_intraday_trader.py:231-238`）。
- 门槛偏低：`TWELVE_MIN_CONFIDENCE = 0.45`（`jarvis_paper_trader.py:488`），且缺仓位建议时兜底 10%（`:489`）。

**风险场景（大白话）**：某天 7 个币同时给出 bullish 共识（大盘普涨时很常见，相关性≈1），自动开满 7 仓等于满仓押同一个方向；回调时 7 仓同时止损，再过 15 分钟共识未翻转又开回来，连环挨打。

**改进方案**：① `open_from_twelve` 仓位走 `evaluate_guardrails` 同款封顶（或至少 clamp 到 `max_position_pct` + 组合风险缩仓）；② 加 `twelve_max_open_positions`（建议默认 3）与 `twelve_cooldown_min`（建议默认 ≥1 个主周期）；③ 多币同向时按相关性折减总敞口（已有 `max_effective_pct` 旋钮，`jarvis_config.py:67`，接进 twelve 路径）。

**预期效果**：自动路径与手动路径同一套红线，杜绝相关性满仓与自动报复性交易。 **成本：M**。

## P1 · 应该修（防护完整性）

### P1-1 熔断一键复位无确认，冷静期可被配置归零

**证据**：`POST /api/circuit-breaker/reset` 无任何确认参数（`jarvis_dashboard.py:546-554`）；对比冷静期提前解锁强制 `{"confirm": true}`（`jarvis_dashboard.py:582-591`）。`reset()` 不清冷静期（`jarvis_circuit_breaker.py:282-292`，设计正确），但 `cooldown_hours` 可配 0=禁用（`jarvis_circuit_breaker.py:106-112`、`jarvis_config.py:150`）——归零后 reset 即完全解锁。
**大白话**：熔断就是「输急眼时按住你手」的东西，一键就能解开等于没按住。
**改进**：reset 也要求 `confirm:true` + 显示当日亏损归因；`cooldown_hours` 下限钳到 ≥1h（clamp 层实现）。 **成本：S**。

### P1-2 连亏递增防护缺失（没有「亏一笔减半仓」）

**证据**：intraday 有连亏 N 笔停机（默认 3，`jarvis_config.py:95`、`jarvis_intraday_trader.py:390-395`）——是「全停」不是「递减」；twelve 与手动路径无任何连亏限制（`jarvis_paper_trader.py` 全文无连亏计数）。
**大白话**：连亏时最需要的是自动越打越小，而不是等亏满 3 笔一刀全停或根本不管。
**改进**：新增连亏敞口衰减：连亏 k 笔 → 后续开仓名义 × 0.5^min(k,3)，盈利一笔恢复一档；三条开仓路径共用（executor/twelve/intraday）。 **成本：M**。

### P1-3 无波动率自适应仓位（高波动不降仓）——关联已讨论的 EWMA σ 方案

**证据**：sizing 仅 fixed/kelly 两种（`jarvis_sizing.py:61-93`），kelly 输入是静态胜率/赔付；ATR 只用于 intraday 止损距离（`jarvis_config.py:91-92`），无任何「σ 升高 → 仓位自动缩」逻辑。
**大白话**：行情疯的时候（波动率翻倍）同样的仓位实际风险翻倍，系统不会自动帮你踩刹车。
**改进**：接 EWMA σ（如 λ=0.94 日频）做仓位缩放：`scale = min(1, target_σ/当前σ)`，套在所有开仓路径的名义计算后；与 P1-2 衰减相乘。 **成本：M**（σ 计算可复用现有 K 线取数）。

### P1-4 未配 agent_token 时权益不计持仓市值，熔断回撤口径失真

**证据**：`jarvis_paper_trader.py:919` — `price = latest_price(...) if cfg.get("agent_token") else None` → 无 token 时持仓市值恒 0，`equity = cash + frozen`（`:931`），开仓扣现金即表现为「权益骤降」；该 equity 正是回撤熔断数据源（`jarvis_circuit_breaker.py:196-207`）。
**大白话**：没配网关 token 的用户，开两笔仓系统就以为你回撤 20%，熔断乱触发（或 peak 混乱后失效）。
**改进**：取价失败/无 token 时用 entry_price 兜底估市值，并在 stats 输出标记 `equity_degraded=true`。 **成本：S**。

### P1-5 下单确认层仅前端提示，后端无硬校验；无频繁交易闸

**证据**：`min_rr_warning=1.5`「黄色警告（不拦单）」、`plan_confirm_enabled` 均为前端确认层旋钮（`jarvis_config.py:127-129`）；后端 `/api/orders/place` 只过熔断门禁，无 RR/追高/频次校验（`jarvis_dashboard.py:1001-1031`）；`place_limit_order` 本身无当日下单次数限制（`jarvis_wallet.py:210-248`）。
**大白话**：前端提示可以被无视，也拦不住脚本/接口直接下单；一天手滑下 50 单没人拦你。
**改进**：后端加软闸：当日新开仓次数 > N（默认 10）或 RR < 阈值时要求请求带 `force=true` 才放行（仍不禁止，但强制「过一道脑子」）；频次进 jarvis_config。 **成本：S/M**。

## P2 · 可以修（口径与体验）

### P2-1 限价撮合用单点现价轮询，插针成交口径不稳定
**证据**：`match_limit_orders` 用 `latest_price` 单点判定（`jarvis_paper_trader.py:798-806`），轮询间隔内的插针（wick）触价会漏成交；反之被判成交的价格真实场景可能只是瞬间刺穿。
**改进**：撮合改用自上次撮合以来的 1m K 线 high/low 判定触价，成交价取 max(limit, next_open) 保守口径。 **成本：M**。

### P2-2 爆仓价分层表仅 BTC 内置官方档
**证据**：`BINANCE_MM_TIERS` 只有 BTC（`jarvis_position_calc.py:49-59`），其余币用 MMR 1% 保守档（`:63-65`）。方向是保守的（爆仓价估得更近），可接受；但山寨高杠杆时误差大。
**改进**：补 ETH/SOL 等 watchlist 币官方档，或启动时从交易所 API 拉取缓存。 **成本：S**。

### P2-3 熔断门禁「异常=放行」的 fail-open 策略应显式告知
**证据**：`guard_new_order` evaluate 失败放行（`jarvis_circuit_breaker.py:363-367`）；`/api/orders/place` 门禁异常 pass（`jarvis_dashboard.py:1007-1014`）；intraday 同（`jarvis_intraday_trader.py:257-258`）。paper-only 下合理，但若未来接实盘必须翻转为 fail-closed。
**改进**：配置项 `cb_fail_mode`（paper 默认 open，live 强制 closed）+ 放行时 UI 显示「风控降级中」。 **成本：S**。

### P2-4 intraday 平仓顺序在同一根 bar 内「先止损后止盈」偏悲观但未文档化
**证据**：`_maybe_close` 固定顺序 stop→take→time→signal（`jarvis_intraday_trader.py:215-228`）。单点现价下无歧义，但若改 K 线判定（P2-1）需明确同 bar 双触规则。
**改进**：文档化口径；回测侧同口径对齐。 **成本：S**。

## 小白三条铁律（从本次发现提炼）

1. **模拟胜率先打八折再谈实盘**——当前模拟盘零手续费零滑点（P0-2），真实交易每笔来回至少 0.1~0.2% 摩擦；模拟里勉强赚钱的策略，实盘大概率亏。修复前，把模拟战绩当「上限」而不是「预期」。
2. **做空单别信界面上的浮盈浮亏**——修复 P0-1 前，空单的权益曲线和熔断保护都是反的；只认平仓后的「已实现盈亏」，开空时手动在计算器里核一遍爆仓价。
